Exam Decision Key
If an exam scenario requires instant startup time with minimal setup, choose Starter Pools. If it requires exact memory allocation, explicit executor counts, or specific node sizes, choose Custom Pools.

Microsoft|Session Concurrency
High Concurrency
Concurrency Model
High Concurrency &
Session Sharing
Share active Spark clusters across multiple notebooks to eliminate startup delays and maximize Capacity Unit (CU) efficiency.
🔀
Shared Spark Sessions
Multiple notebooks attach to an existing active session in seconds instead of spinning up isolated clusters per notebook.
💲
CU Cost Savings
Prevents idle capacity consumption by reusing executor memory and CPU across concurrent data engineering workloads.
🔒
User Security Isolation
Despite sharing compute nodes, Microsoft Fabric maintains user-level security boundaries and variable isolation across sessions.
🔄
Pipeline Integration
Data Factory pipelines executing multiple notebook activities in parallel reuse the same high concurrency session automatically.
✅ When to Enable High Concurrency Mode
• Interactive team development in shared workspaces.
• Pipelines that trigger 5+ PySpark Notebooks concurrently.
• Workspaces seeking to reduce overall Fabric Capacity Unit (CU) throttle risks.

Microsoft|Performance Engine
Native Engine
Performance Boost
Native Execution Engine
(Up to 4x Faster)
Replaces standard Java JVM execution with a C++ Velox vectorized processing engine directly inside Fabric.
⚡
C++ Vectorized Processing
Executes columnar operations directly on CPU SIMD instructions, bypassing JVM garbage collection overhead.
📈
4x Speedup on Queries
Dramatically speeds up heavy aggregations, joins, string operations, and Delta table scans.
💻 How to Enable in Spark Session / Notebook
# Enable Native Execution Engine in PySpark spark.conf.set("spark.native.enabled", "true") # Verify engine execution status in Physical Plan df.explain() # Look for 'NativeScan' or 'Velox' operators
🎯
Exam Tip
When an exam scenario asks how to optimize long-running PySpark transformation jobs without changing code or increasing pool size, enabling the Native Execution Engine is the top answer.

Microsoft|Library Governance
Library Management
Library Governance
Environments vs
Inline Libraries
Managing Python/PyPI dependencies efficiently across development workspaces.
📦
Environment Items (Recommended)
Create reusable Environment items containing PyPI/Conda YAML specs. Attach to workspaces or specific notebooks for governance.
📝
Inline %pip Commands
Use %pip install library_name at the top of a notebook for temporary ad-hoc testing. Installs only for the current session.
💻 Example: Conda Environment YAML Definition
name: fabric_data_engineering_env channels: - conda-forge dependencies: - python=3.10 - pip: - azure-storage-blob==12.19.0 - delta-spark==3.1.0
ⓘ
Best Practice
Avoid using inline %pip install in production scheduled pipelines because it adds installation latency on every pipeline run. Pre-publish dependencies into an Environment Item.

🎓 What to Look for in Spark UI (Exam Scenarios)
• Spill (Disk) & Spill (Memory): Indicates memory pressure; increase executor size in Custom Pool.
• Data Skew: One task taking 90% of job duration; apply repartitioning or skew hints.
• Stragglers: Uneven partition distribution across executors.
