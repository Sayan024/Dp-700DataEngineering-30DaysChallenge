Data Engineering›DP-700›Pool Sizing
Pool Architecture
Starter Pools vs Custom Pools
Choosing the right compute pool type directly impacts job startup latency, scalability, and capacity costs.
⚡ Starter Pools (Default Configuration)
⏱
Startup Time
~5 to 10 Seconds (Pre-warmed background nodes)
🛠
Management
Fully Managed by Fabric (Zero admin overhead)
📈
Auto-Scale
Dynamic Node Allocation based on load
💻
Best Used For
Ad-hoc EDA, Quick Notebooks & Interactive Queries
⚙ Custom Pools (Dedicated Sizing)
⏱
Startup Time
~2 to 3 Minutes (Dedicated Cold Start)
📏
Node Sizing
Small (4 vCore) to XX-Large (64 vCore / 512 GB)
⏸
Auto-Pause
Configurable idle timeout (Default 2 mins)
🚀
Best Used For
Heavy Batch ETL & Strict Memory Allocation
💡
Exam Decision Key
If an exam scenario requires instant startup time with minimal setup, choose Starter Pools. If it requires exact memory allocation, explicit executor boundaries, or specific node sizes, choose Custom Pools.

Microsoft|Session Concurrency
High Concurrency
Data Engineering›DP-700›High Concurrency
Concurrency Model
High Concurrency & Session Sharing
Share active Spark clusters across multiple notebooks to eliminate startup delays and maximize Capacity Unit (CU) efficiency.
🔀
Shared Spark Sessions
Multiple notebooks attach to an existing active session in seconds instead of spinning up isolated clusters.
💲
CU Cost Savings
Prevents idle capacity consumption by reusing executor memory and CPU across concurrent engineering workloads.
🔒
User Security Isolation
Microsoft Fabric maintains user-level security boundaries and variable isolation across shared sessions.
🔄
Pipeline Reuse
Data Factory pipelines executing multiple notebook activities in parallel reuse the same session automatically.
✅ When to Enable High Concurrency Mode in Workspace Settings
• Interactive Team Development: Multiple data engineers working in shared workspaces simultaneously.
• Parallel Notebook Orchestration: Pipelines that trigger 5+ PySpark Notebooks concurrently.
• Capacity Throttling Mitigation: Workspaces seeking to eliminate burst CU consumption and queuing delays.

Microsoft|Performance Engine
Native Engine
Data Engineering›DP-700›Native Velox
Performance Boost
Native Execution Engine (Up to 4x Faster)
Replaces standard Java JVM execution with a C++ Velox vectorized processing engine directly inside Fabric.
⚡
C++ Vectorized Processing
Executes columnar operations directly on CPU SIMD instructions, bypassing JVM garbage collection overhead.
📈
4x Speedup on Queries
Dramatically speeds up heavy aggregations, joins, string operations, and Delta table scans.
💻 How to Enable in PySpark Session
# Enable Native Execution Engine in PySpark spark.conf.set("spark.native.enabled", "true") # Verify engine execution status in Physical Plan df = spark.read.format("delta").load("Tables/sales") df_filtered = df.filter(df["total_amount"] > 1000).groupBy("region").count() df_filtered.explain() # Look for 'NativeScan' or 'Velox'
🎯
Exam Tip
When an exam scenario asks how to optimize long-running PySpark transformation jobs without changing code or increasing pool size, enabling the Native Execution Engine is the top answer.

Microsoft|Library Governance
Library Management
Data Engineering›DP-700›Environment Items
Library Governance
Environments vs Inline Libraries
Managing Python/PyPI dependencies efficiently across development workspaces.
📦
Environment Items (Recommended)
Create reusable Environment items containing PyPI/Conda YAML specs. Attach to workspaces or specific notebooks for governance.
📝
Inline %pip Commands
Use %pip install library_name at the top of a notebook for temporary ad-hoc testing. Installs only for the current session.
💻 Example: Conda Environment YAML Definition
name: fabric_data_engineering_env channels: - conda-forge dependencies: - python=3.10 - pip: - azure-storage-blob==12.19.0 - delta-spark==3.1.0
ⓘ
Best Practice
Avoid using inline %pip install in production scheduled pipelines because it adds installation latency on every pipeline run. Pre-publish dependencies into an Environment Item.

• Spill (Memory) & Spill (Disk): Indicates memory pressure; increase executor size in Custom Pool.
